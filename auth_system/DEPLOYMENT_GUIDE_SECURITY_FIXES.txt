STEP-BY-STEP DEPLOYMENT:

─────────────────────────────────────────────────────────────────────────────
STEP 1: Backup Your Database
─────────────────────────────────────────────────────────────────────────────

COMMAND (MySQL):
  mysqldump -u DB_USER -p DB_NAME > backup_$(date +%Y%m%d_%H%M%S).sql

COMMAND (Docker):
  docker exec vaultly_db mysqldump -u root -pROOT_PASSWORD vaultly > backup.sql

Verify backup created successfully before proceeding.


─────────────────────────────────────────────────────────────────────────────
STEP 2: Deploy Updated PHP Files
─────────────────────────────────────────────────────────────────────────────

Copy these files to your server (replacing existing versions):

  ✓ includes/db.php
  ✓ api/vault.php
  ✓ register.php
  ✓ api/health.php (NEW FILE)

Command (if using rsync):
  rsync -av includes/db.php api/vault.php register.php api/health.php \
    user@server:/path/to/vaultly/

Verify files deployed:
  ls -la includes/db.php api/vault.php register.php api/health.php


─────────────────────────────────────────────────────────────────────────────
STEP 3: Database Migration (Automatic)
─────────────────────────────────────────────────────────────────────────────

NO MANUAL STEPS REQUIRED. The 'version' column is created automatically when
includes/db.php loads.

To verify migration:
  mysql -u DB_USER -p DB_NAME -e "SHOW COLUMNS FROM vault_entries;"

Look for the 'version' column in output. It should show INT(11) DEFAULT 1.

If not present after loading PHP, run manually:
  mysql -u DB_USER -p DB_NAME -e "ALTER TABLE vault_entries ADD COLUMN version INT DEFAULT 1;"


─────────────────────────────────────────────────────────────────────────────
STEP 4: Test Health Check Endpoint
─────────────────────────────────────────────────────────────────────────────

LOCAL TEST:
  php api/health.php

NETWORK TEST:
  curl http://localhost/api/health.php
  curl https://yourdomain.com/api/health.php

EXPECTED RESPONSE (healthy):
  {
    "ok": true,
    "status": "healthy",
    "checks": { "database": true, "disk_free": true },
    "uptime": 14400,
    "timestamp": "2025-01-15T10:30:45+00:00"
  }

RESPONSE CODES:
  HTTP 200: Healthy (all checks passed)
  HTTP 503: Unhealthy (database down or disk full)


─────────────────────────────────────────────────────────────────────────────
STEP 5: Apply XSS Fixes to vault-ui.js
─────────────────────────────────────────────────────────────────────────────

IMPORTANT: This requires manual code edits to js/vault-ui.js

Guidance file: XSS_FIXES_VAULT_UI.txt (in your repo)

Quick summary of changes:

1. Add escapeHTML() function to vault-ui.js IIFE:
   function escapeHTML(str) {
       const div = document.createElement('div');
       div.appendChild(document.createTextNode(str || ''));
       return div.innerHTML;
   }

2. Replace all esc(decrypted_data) with escapeHTML(decrypted_data) in:
   - renderEntries() — entry titles, subtitles
   - viewEntry() — modal fields
   - field() — form field values
   - populateFolderSelect() — folder names
   - All other user-controlled data rendering

Detailed line-by-line fixes: see XSS_FIXES_VAULT_UI.txt


─────────────────────────────────────────────────────────────────────────────
STEP 6: Configure Environment Variables
─────────────────────────────────────────────────────────────────────────────

Create or update .env file:

  # Security
  APP_DEBUG=false              # Production: never true
  HTTPS_ENFORCE=true           # Require HTTPS (once SSL cert installed)
  SESSION_TIMEOUT=3600         # 1 hour session lifetime
  
  # Database
  DB_HOST=localhost
  DB_USER=vaultly_user
  DB_PASS=<strong_random_password>
  DB_NAME=vaultly
  
  # Optional
  BACKUP_PATH=/backups/vaultly
  LOG_LEVEL=info               # For production logging

Verify .env is NOT checked into version control (add to .gitignore).


─────────────────────────────────────────────────────────────────────────────
STEP 7: Enable HTTPS + SSL Certificate
─────────────────────────────────────────────────────────────────────────────

In .htaccess, uncomment these lines once SSL cert is installed:

  # HTTPS enforcement
  <IfModule mod_rewrite.c>
      RewriteEngine On
      RewriteCond %{HTTPS} off
      RewriteRule ^(.*)$ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]
  </IfModule>

  # Also uncomment in PHP session hardening:
  # php_flag session.cookie_secure on

CERTIFICATE INSTALLATION (Let's Encrypt example):
  certbot certonly --webroot -w /path/to/vaultly -d yourdomain.com
  certbot install
  systemctl restart apache2


─────────────────────────────────────────────────────────────────────────────
STEP 8: Configure Docker (if applicable)
─────────────────────────────────────────────────────────────────────────────

Update Dockerfile HEALTHCHECK:

  FROM php:8.1-apache
  
  # ... existing config ...
  
  HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
    CMD curl -f http://localhost/api/health.php || exit 1

Update docker-compose.yml:

  services:
    web:
      build: .
      ports:
        - "80:80"
        - "443:443"
      environment:
        APP_DEBUG: "false"
        DB_HOST: db
        DB_USER: vaultly_user
        DB_PASS: ${DB_PASS}
        DB_NAME: vaultly
      depends_on:
        db:
          condition: service_healthy
      healthcheck:
        test: ["CMD", "curl", "-f", "http://localhost/api/health.php"]
        interval: 30s
        timeout: 5s
        retries: 3

Deploy:
  docker-compose up -d --build
  docker-compose logs web


─────────────────────────────────────────────────────────────────────────────
STEP 9: Verify Secure Cookie Headers
─────────────────────────────────────────────────────────────────────────────

Check that cookies are secure:

  curl -i https://yourdomain.com/login.php | grep -i set-cookie

EXPECTED OUTPUT (all flags present):
  Set-Cookie: vaultly_session=...; Path=/; HttpOnly; Secure; SameSite=Strict

If missing flags, verify:
  1. APP_DEBUG=false in production
  2. HTTPS is enabled (Secure flag requires HTTPS)
  3. .htaccess is loaded (check apache2ctl -t)


─────────────────────────────────────────────────────────────────────────────
STEP 10: Test Vault Concurrent Edit (Race Condition Fix)
─────────────────────────────────────────────────────────────────────────────

SCENARIO: Two clients edit the same entry simultaneously

STEPS:
  1. Open https://yourdomain.com in Browser A
  2. Open https://yourdomain.com in Browser B (different user or incognito)
  3. Both login to vault
  4. Both view the same entry
  5. In Browser A: Edit entry title, save
  6. In Browser B: Edit entry title, save
  7. Browser B should receive HTTP 409 with version conflict

EXPECTED RESPONSE (Browser B):
  {
    "error": "Version conflict: entry was modified. Refresh and try again.",
    "current_version": 2,
    "your_version": 1
  }

✓ This confirms optimistic locking is working.


─────────────────────────────────────────────────────────────────────────────
STEP 11: Test Database Transactions
─────────────────────────────────────────────────────────────────────────────

SCENARIO: Verify atomic registration (either succeeds completely or fails)

STEPS:
  1. Navigate to /register.php
  2. Create new account
  3. Check database:
     mysql> SELECT * FROM users WHERE email = 'test@example.com';
     mysql> SELECT * FROM security_logs WHERE details LIKE '%test@example.com%';
  4. Both records should exist (or neither if registration failed)

VERIFY TRANSACTION:
  mysql> SHOW TABLES;  # Verify vault_entries has 'version' column


─────────────────────────────────────────────────────────────────────────────
STEP 12: Enable Logging & Monitoring
─────────────────────────────────────────────────────────────────────────────

Monitor security_logs for suspicious activity:

  mysql> SELECT event_type, COUNT(*) as count FROM security_logs 
         GROUP BY event_type 
         ORDER BY count DESC;

Set up alerts for:
  - rate_limit_exceeded: Brute force attempts
  - csrf_violation: CSRF attacks
  - registration_error: Unexpected registration failures
  - invalid_vault_action: Unauthorized API calls

Optional: Ship logs to external service (ELK, Datadog, CloudWatch):
  Modify logSecurityEvent() in includes/db.php to post to syslog

SUPPORT & TROUBLESHOOTING
═══════════════════════════════════════════════════════════════════════════════

Issue: version column not appearing
  → Check PHP output for errors
  → Manually run: ALTER TABLE vault_entries ADD COLUMN version INT DEFAULT 1;

Issue: Health check returns unhealthy
  → Check database connectivity: mysql -u user -p -e "SELECT 1;"
  → Check disk space: df -h
  → Check error_log: tail /var/log/apache2/error.log

Issue: Concurrent edit test doesn't produce 409
  → Verify client is sending 'version' in request body
  → Check vault.php is deployed (not old version)
  → Monitor database logs

Issue: Session cookies missing HttpOnly flag
  → Ensure APP_DEBUG=false in production
  → Verify HTTPS enabled (Secure flag requires it)
  → Check .htaccess php_flag session.cookie_httponly on
